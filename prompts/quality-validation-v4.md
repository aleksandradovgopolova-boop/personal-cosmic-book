# Контроль качества v4

## Уровень 1: программные проверки без LLM

### Вход и схемы

- JSON Schema validation каждого этапа.
- Все `evidence_refs`, `pattern_ids`, `claim_ids` существуют.
- Нет ссылок на запрещённые low-quality evidence.
- Присутствуют ровно 10 обязательных разделов с правильными ID и порядком.
- Проверка длины и `finish_reason`.
- `requested_theme` входит в `auto | earth | water | air | fire`.
- `resolved_theme` входит в `earth | water | air | fire`.
- `visual_assignment.algorithm_version` известен приложению.
- При `resolution_reason = user_choice` resolved theme совпадает с requested theme.

### Архитектура десяти разделов

- Ровно 10 основных разделов; обложка, оглавление, вступление и дисклеймер не считаются разделами.
- `01_main_theme`, `08_life_patterns` и `10_next_chapter` последовательно раскрывают главный запрос, не повторяя один ответ дословно.
- Астрология, нумерология, матрица и Саджу не превращаются в отдельные главы или четыре последовательных мини-лекции.
- `09_personal_formula` организован вокруг выводов о человеке, а не перечня систем.
- `10_next_chapter` не содержит новых claims.
- Ни один раздел не превышает 7 основных claims.

### Grounding

- Каждый аналитический block имеет минимум один `claim_id`.
- Каждый claim имеет хотя бы один evidence ref.
- Нельзя использовать claims другого owner section как новые.
- `visual_assignment`, название темы и стихия не используются как evidence для текста.
- В тексте нельзя утверждать «твоя стихия — ...» только на основании оформления.

### Повторы

- точные n-gram повторы;
- повтор одинаковых советов;
- повтор открывающих фраз;
- embeddings similarity между абзацами и section summaries;
- лимит на повтор ключевого тезиса вне owner section.

### Язык и безопасность

Отклонять или отправлять на repair предложения с конструкциями:

- `предсказание` в значении факта будущего;
- `гарантирует`, `гарантия`, `точно будет`;
- `судьба решила`, `звёзды говорят`, `предначертано`;
- `научно доказано`, `диагноз`;
- `вам суждено`, `неизбежно`, `обязательно произойдёт`.

Не делать механическую замену стоп-слов. Регенерировать нарушающее предложение.

### Element assignment

- Score рассчитывается кодом по версионированному алгоритму.
- Используются только разрешённые объекты и веса.
- Low/excluded placements не учитываются.
- Ascendant учитывается только при `angles_allowed = true`.
- Ambiguous Moon не угадывается.
- Tie-break повторяем и покрыт unit tests.
- Hash fallback стабилен для одного `book_id`.
- Саджу, нумерология и матрица не участвуют в назначении четырёхстихийной темы.
- Изменение текста или модели не меняет `resolved_theme`.

### Дизайн и HTML

Проверяется после renderer:

- используется только `earth | water | air | fire`;
- модель не передала CSS, hex-цвета, шрифты, symbol или layout instructions;
- один контент во всех четырёх темах сохраняет одинаковую структуру;
- contrast и переносы проверены для каждой темы;
- cosmic signature создана кодом;
- parser validation;
- обязательные anchors;
- no script injection;
- accessibility smoke test;
- responsive smoke test;
- print/PDF smoke test;
- visual regression для обложки, оглавления, типового раздела, практики и финального письма;
- ни одна тема визуально не выглядит как более дешёвый или урезанный вариант.

## Уровень 2: semantic QA моделью

Рекомендуется DeepSeek V4 Pro в thinking mode. Модель не переписывает книгу целиком, а возвращает список проблем и точечные patches.

`presentation` и `visual_assignment` не передаются semantic QA модели: оформление не является основанием для оценки смысла.

### System prompt

Ты — строгий редактор и контролёр качества персональной книги.

Проверь текст только по предоставленным паттернам, blueprint, claim ledger и evidence catalog. Не добавляй собственных трактовок.

Найди:

1. утверждения без основания;
2. выдуманные биографические факты;
3. повторы между десятью разделами;
4. противоречия;
5. банальные и универсальные формулировки;
6. фатализм и опасные обещания;
7. рекомендации, не связанные с анализом;
8. разделы, которые не отвечают своей центральной задаче;
9. потерю главного запроса пользователя;
10. стилистические сбои и искусственную эзотерическую патетику.

Верни только JSON object. Для каждой проблемы предложи минимальную замену. Не переписывай корректные части.

### Output schema

```json
{
  "status": "pass | repair | reject",
  "scores": {
    "grounding": 0,
    "personalization": 0,
    "non_repetition": 0,
    "coherence": 0,
    "tone": 0,
    "safety": 0
  },
  "issues": [
    {
      "severity": "critical | major | minor",
      "section_id": "05_love",
      "block_index": 1,
      "issue_type": "unsupported_claim | repetition | contradiction | cliche | fatalism | off_scope",
      "quote": "Проблемный фрагмент",
      "reason": "Почему это проблема",
      "replacement": "Минимальная исправленная версия",
      "evidence_refs": []
    }
  ],
  "missing_claim_ids": [],
  "duplicated_claim_ids": []
}
```

## Retry policy

1. Schema error — один автоматический retry с сообщением валидатора.
2. Unsupported evidence — раздел регенерируется; неверную ссылку нельзя просто удалить.
3. Minor issues — применить patches.
4. Major issues — регенерировать только проблемный block.
5. Critical issues — регенерировать раздел на более сильной модели.
6. Ошибка theme assignment не вызывает LLM retry; исправляется код и тесты.
7. Не более двух повторов одного модельного этапа; затем зафиксировать bad case.
