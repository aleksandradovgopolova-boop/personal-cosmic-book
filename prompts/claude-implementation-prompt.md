# Готовый промт для Claude

```text
Ты работаешь в существующем репозитории проекта Personal Cosmic Book. Не создавай новый проект и не ограничивайся рекомендациями: изучи текущую реализацию и внеси изменения в код, схемы, renderer, UI и тесты.

Перед началом:
1. Изучи структуру репозитория, текущий pipeline генерации книг, API-контракты, модели данных, renderer HTML/PDF, UI выбора оформления и тесты.
2. Изучи приложенный архив `personal-cosmic-book-prompts-v5` и используй его как целевую продуктовую и техническую спецификацию.
3. Сначала составь короткий план изменений и список найденных файлов, затем сразу приступай к реализации.
4. Не пересчитывай астрологию, нумерологию, матрицу судьбы и Саджу через LLM. Сохрани существующие локальные расчёты.
5. Не задавай вопросы, если ответ можно получить из репозитория или спецификации. При неоднозначности выбирай наиболее безопасное обратимое решение и фиксируй допущение.

Цель обновления

A. Основная книга должна содержать ровно 10 смысловых разделов:
1. `01_main_theme` — Главная тема твоей книги
2. `02_inner_world` — Твой внутренний мир
3. `03_strengths` — Твои сильные стороны
4. `04_inner_tensions` — Внутренние противоречия
5. `05_love` — Любовь и близость
6. `06_realization` — Реализация, дело и деньги
7. `07_outer_expression` — Как ты проявляешься в мире
8. `08_life_patterns` — Повторяющиеся сценарии и точки роста
9. `09_personal_formula` — Твоя персональная формула
10. `10_next_chapter` — Твой следующий шаг

Обложка, оглавление, вступление и дисклеймер не считаются разделами. Удали зависимости от старой структуры из 19 разделов: схемы, prompt templates, счётчики, оглавление, anchors, тестовые fixtures и UI.

B. Дизайн должен быть сокращён до одной дизайн-системы и четырёх фиксированных тем, связанных со стихиями:
- `earth` — «Земля · Корни»;
- `water` — «Вода · Глубина»;
- `air` — «Воздух · Дыхание»;
- `fire` — «Огонь · Искра».

Не генерируй уникальный CSS, layout, палитру или обложку через LLM. LLM отвечает только за текст. Renderer отвечает за визуальную часть.

C. Пользовательский выбор темы

В presentation preferences добавь:

`requested_theme: auto | earth | water | air | fire`

В UI покажи пять карточек:
- «Подобрать по карте»;
- «Земля · Корни» — спокойное и собранное оформление;
- «Вода · Глубина» — мягкое и глубокое оформление;
- «Воздух · Дыхание» — светлое и просторное оформление;
- «Огонь · Искра» — тёплое и выразительное оформление.

Добавь пояснение:
«Стихия влияет только на оформление книги. Она не меняет расчёты и содержание.»

Не маркируй темы как женские/мужские, сильные/слабые, точные/неточные или разные тарифы.

D. Автоматическое назначение темы

Реализуй отдельный deterministic module, не связанный с LLM.

Он должен возвращать:

{
  algorithm_version: "astrology-four-elements-v1",
  element_scores: { earth, water, air, fire },
  dominant_element,
  resolved_theme,
  resolution_reason: "user_choice | dominant_element | hash_fallback",
  tie_breaker,
  warnings
}

Веса объектов:
- Sun = 3
- Moon = 3
- Ascendant = 2, только если angles разрешены
- Mercury = 2
- Venus = 2
- Mars = 2
- Jupiter = 1
- Saturn = 1

Соответствие знаков:
- fire: Aries, Leo, Sagittarius
- earth: Taurus, Virgo, Capricorn
- air: Gemini, Libra, Aquarius
- water: Cancer, Scorpio, Pisces

Правила:
1. Low/excluded placements не учитывать.
2. Ascendant учитывать только при `angles_allowed = true`.
3. Если Moon ambiguous из-за неизвестного времени, не угадывать и не учитывать её без отдельного явного правила.
4. Если пользователь выбрал конкретную тему, она имеет приоритет.
5. В auto выбрать максимальный score.
6. При равенстве: Sun среди лидеров → Moon среди лидеров → stable hash от book_id среди лидеров.
7. Если данных недостаточно, stable hash от book_id по четырём темам.
8. Одинаковый input + algorithm_version должен всегда давать одинаковый результат.
9. Не использовать Саджу для этого выбора: его Wood/Fire/Earth/Metal/Water нельзя произвольно сводить к четырём стихиям.
10. Не использовать нумерологию и матрицу судьбы для выбора дизайна.

E. Дизайн-токены

Перенеси темы в код как неизменяемые tokens. Используй значения из `design-system-elements-v2.md`.

Одинаковыми для всех тем должны остаться:
- layout;
- сетка;
- типографика;
- компоненты;
- структура обложки;
- структура 10 разделов;
- навигация;
- responsive и print styles.

Меняться могут только:
- tokens;
- фиксированный symbol стихии;
- семейство SVG-pattern;
- декоративные разделители;
- cosmic signature, детерминированная из book_id.

Убери зависимость дизайна от `primary_archetype`. Убери библиотеку 12 архетипических обложек, если она используется только ради визуальной вариативности. Смысловой archetype можно сохранить в аналитике, но он не должен назначать тему.

F. Разделение LLM и renderer

Проверь, что stage 2–4 prompts не получают:
- requested/resolved theme;
- element scores;
- цвета;
- CSS;
- symbol;
- layout instructions.

Модель должна возвращать JSON-контент одного раздела за вызов. HTML/PDF формируется кодом.

Тема не должна влиять на claims, заголовки и психологические выводы. Не допускай фраз типа «поскольку твоя стихия — Вода» только из-за выбранного оформления.

G. Миграция старых тем

Добавь временные backend aliases:
- `cosmic_night` → `water`;
- `solar_dawn` → `fire`;
- `lunar_mist` → `air`.

Новые записи сохраняй только как `earth | water | air | fire`.
Для старого `auto`, если расчётные данные доступны, выполни новый element assignment.
Не регенерируй текст уже созданных книг только из-за смены темы.
После нормализации renderer и публичный API не должны возвращать legacy IDs.

H. Дополнения

Прогноз и совместимость по умолчанию наследуют resolved theme основной книги и используют ту же design system. Не создавай отдельные стили для дополнений. Если дополнение создаётся отдельно, используй тот же assignment module.

I. Тесты и QA

Добавь или обнови:
1. JSON Schema tests для 10 разделов и theme enums.
2. Unit tests для element score, exact wins, ties, missing birth time, ambiguous Moon, user override и hash fallback.
3. Migration tests для трёх legacy theme IDs.
4. Renderer tests для одинаковой структуры во всех четырёх темах.
5. Contrast/accessibility checks.
6. Responsive smoke tests.
7. HTML/PDF print smoke tests.
8. Visual regression минимум для обложки, оглавления, типового раздела, блока практики и финального письма во всех четырёх темах.
9. Test, что изменение LLM-модели или текста не меняет resolved theme.
10. Test, что theme data не попадает в model payload stage 2–4.

J. Критерии готовности

Работа считается завершённой, когда:
- нигде в активном pipeline не осталось ожидания 19 разделов;
- книга стабильно содержит ровно 10 разделов;
- UI предлагает auto + 4 темы;
- auto assignment полностью deterministic;
- LLM не генерирует дизайн и не выбирает тему;
- renderer поддерживает ровно 4 новые theme IDs;
- старые theme IDs корректно мигрируют;
- все проверки и тесты проходят;
- README/technical docs в репозитории обновлены;
- в финальном отчёте перечислены изменённые файлы, принятые решения, выполненные тесты и оставшиеся риски.

Не делай косметический patch поверх старой архитектуры. Удали или мигрируй устаревшие ветки, чтобы в коде существовал один понятный source of truth.
```
