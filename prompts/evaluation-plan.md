# План сравнительного теста DeepSeek, Kimi и стихийных тем

## Golden set для текста

Собрать минимум 20 обезличенных профилей:

- 5 с точным временем рождения;
- 5 без времени рождения;
- 5 с выраженными противоречивыми сигналами;
- 5 с разными главными запросами: отношения, работа, самооценка, деньги, переходный период.

Не использовать только «красивые» профили. Добавить реальные bad cases.

## Сравниваемые конфигурации

1. DeepSeek V4 Flash end-to-end.
2. DeepSeek V4 Pro end-to-end.
3. Hybrid: Pro для synthesis/blueprint/QA + Flash для текста.
4. Kimi K2.6 по тому же pipeline.

## Автоматические метрики текста

- schema pass rate;
- unknown evidence reference rate;
- missing/wrong section rate для структуры из 10 разделов;
- repeated claim rate;
- unsupported biographical claim rate;
- safety violation rate;
- regeneration rate;
- median latency;
- input/output tokens;
- стоимость успешной книги, включая retries.

## Человеческая оценка текста

Каждую книгу оценивают вслепую минимум два человека по шкале 1–5:

1. Узнаваемость и конкретность.
2. Глубина без банальностей.
3. Связность всей книги.
4. Отсутствие повторов.
5. Практическая полезность.
6. Теплота и качество русского языка.
7. Отсутствие выдуманной биографии.
8. Желание сохранить или подарить книгу.

## Главная продуктовая метрика качества

```text
стоимость одной книги, прошедшей QA и получившей среднюю человеческую оценку >= 4.2
```

## Решение о модели

- Если Hybrid DeepSeek уступает Kimi менее чем на 0.2 балла, выбирать DeepSeek.
- Если Kimi выигрывает прежде всего стилем, тестировать его только для stylistic rewrite проблемных разделов.
- Не менять основную модель по единичному красивому примеру.

## Golden set для element assignment

Добавить минимум 24 технических fixture:

- по 4 профиля с явным доминированием каждой стихии;
- 4 tie cases;
- 4 профиля без точного времени;
- 2 профиля с ambiguous Moon;
- 2 профиля с недостаточными данными и hash fallback;
- legacy themes для проверки миграции.

Проверять повторяемость результата при одинаковом `book_id` и версии алгоритма.

## Visual QA четырёх тем

Для каждой контрольной книги рендерить:

- `earth`;
- `water`;
- `air`;
- `fire`.

Проверять:

- одинаковую структуру и page breaks;
- отсутствие переполнений;
- contrast;
- корректность PDF;
- узнаваемость бренда;
- отсутствие ощущения, что одна тема премиальная, а другие второстепенные;
- читаемость длинных имён и заголовков;
- обложку, оглавление, типовой раздел, практику и письмо себе;
- корректное отображение символа и cosmic signature.

## Пользовательский тест выбора темы

Сравнить два сценария:

1. Пользователь выбирает тему сам.
2. Пользователь оставляет «Подобрать по карте».

Метрики:

- доля ручного выбора;
- доля смены автоматически назначенной темы;
- удовлетворённость оформлением;
- ощущение персональности книги;
- понятность пояснения, что стихия влияет только на дизайн;
- частота ошибочного восприятия темы как «диагноза личности».

Если пользователи массово считают auto-тему психологическим вердиктом, усилить микрокопирайтинг, а не усложнять алгоритм.
