# Детерминированный HTML/PDF renderer v2

## Принцип

Модель не генерирует HTML, CSS, палитру, символ или уникальный визуальный стиль книги. Сервер рендерит валидированные JSON-разделы через один компонентный шаблон и одну дизайн-систему.

Персональность книги создаётся содержанием, именем, заголовком, одной из четырёх стихийных тем и ограниченным детерминированным декором — не новой вёрсткой для каждого человека.

## Структура документа

Обложка, оглавление, короткое редакторское вступление и дисклеймер являются служебными страницами и не входят в число десяти разделов книги.

Основная книга содержит ровно 10 разделов с ID из `book-structure-10-sections.md`.

## Что хранится в коде

- единый HTML skeleton;
- компонентная дизайн-система;
- четыре фиксированные темы оформления;
- CSS tokens;
- шрифты и fallback;
- навигация;
- шаблоны обложки и разделителей;
- четыре фиксированных SVG-символа стихий;
- четыре семейства фоновых SVG-паттернов;
- print styles;
- footer и дисклеймер;
- адаптивность;
- правила page-break;
- экранирование пользовательских данных.

## Разрешённые темы

Renderer принимает только:

```text
earth | water | air | fire
```

Полные токены и визуальные правила находятся в `design-system-elements-v2.md`.

## Выбор темы

1. Renderer использует только `visual_assignment.resolved_theme`.
2. `resolved_theme` формируется серверным assignment module до renderer.
3. Если `requested_theme != auto`, assignment module сохраняет пользовательский выбор.
4. Если `requested_theme = auto`, assignment module использует `element_scores` и правила из `input-data-contract-v3.md`.
5. При отсутствии валидного `resolved_theme` renderer не угадывает тему и не вызывает LLM. Он применяет детерминированный hash fallback и логирует ошибку контракта.
6. Назначение темы не зависит от `primary_archetype`, текста книги и ответа модели.
7. Саджу не преобразуется в четырёхстихийную тему.

## Допустимая персонализация

Для каждой книги разрешено менять только:

- имя на обложке;
- персональный заголовок и подзаголовок;
- одну из четырёх тем;
- соответствующий теме символ;
- детерминированный SVG-узор / cosmic signature из `book_id`;
- короткую подарочную надпись;
- акцентные цитаты из валидированного текста.

Запрещено генерировать новую сетку, новый CSS, новую систему иллюстраций, произвольную палитру или отдельный layout для конкретной книги.

## Renderer context

```json
{
  "book_id": "stable-book-id",
  "resolved_theme": "water",
  "theme_tokens": {},
  "element_symbol": "water",
  "signature_seed": "stable-hash",
  "cover": {
    "name": "Софья",
    "title": "Книга о тебе",
    "subtitle": "...",
    "gift_message": null
  },
  "sections": [],
  "disclaimer": "Материалы носят исключительно развлекательный и информационный характер. Не являются научно обоснованными прогнозами."
}
```

`theme_tokens` загружаются из кода по `resolved_theme`, а не принимаются из API клиента или LLM.

## Миграция legacy themes

При чтении старых записей допускаются только временные aliases:

- `cosmic_night` → `water`;
- `solar_dawn` → `fire`;
- `lunar_mist` → `air`.

Правила:

1. Alias нормализуется на backend до renderer.
2. Новые записи сохраняются только как `earth | water | air | fire`.
3. Для legacy `auto` при наличии расчётных данных выполняется новый element assignment.
4. После миграции aliases удаляются из публичных API и UI.
5. Миграция не должна регенерировать текст книги.

## Обязательные проверки

1. HTML валиден.
2. Все пользовательские строки экранированы.
3. Нет inline script из модельного ответа.
4. Присутствуют ровно 10 основных разделов в правильном порядке.
5. Служебные страницы не размечены как основной раздел.
6. Navigation anchors уникальны.
7. Есть mobile layout и print stylesheet.
8. Есть обязательный footer:

`Материалы носят исключительно развлекательный и информационный характер. Не являются научно обоснованными прогнозами.`

9. Цветовой контраст проверен автоматически для каждой из четырёх тем.
10. Книга корректно открывается без внешнего JavaScript.
11. Ошибка одного раздела не ломает весь документ.
12. Оглавление показывает ровно 10 разделов.
13. Используется только один из четырёх разрешённых `resolved_theme`.
14. Все четыре темы проходят одинаковый visual regression test.
15. Print/PDF-версия не использует сплошную тёмную заливку на внутренних страницах.
16. `secondary` и `metal` не используются для длинного текста без contrast pass.
17. Тема не изменяет текстовые данные и порядок блоков.
18. Повторный renderer run с одинаковым context даёт идентичный HTML, кроме явно разрешённых timestamp-полей.
